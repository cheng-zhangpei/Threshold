## Threshold 安全代理基准测试结果

### 测试环境

| 项目     | 值                            |
| -------- | ----------------------------- |
| OS       | WSL2 (Ubuntu) on Windows      |
| CPU      | Intel (Laptop)                |
| 后端     | boltdb HTTP Server (:8080)    |
| 测试工具 | Go bench 客户端 (tools/bench) |
| 每组重复 | 3 次取中位数                  |
| 每次持续 | 30 秒                         |

因为目前我们的两个场景都是IO密集型的场景，设计到块设备的存储等，所以我们本地也是用一个单机数据库模拟IO密集型应用。但是单机数据库能够体现的内容较为有限，在有条件的情况下换为分布式存储池。

后续所有的指标是5次实验的中位数

> 本地PC配置和测试条件有限，最好要在稍微正式点的环境测试，因为使用本地回环进行测试，数据包没有出网卡，所以下面的测试数据距离真实的网络环境会有巨大的偏差。

目前项目一共三种模式适用于**不同的场景**，我们的测试重点集中在后面两种模式。三种模式的概述如下:

1. http层级的代理，性能最差，但是语义性最强，可以直接在业务层去进行详细拦截规则和用户肖像的配置，适用于强功能，弱性能的场景
2. tcp层，socks5 + grpc，性能中等，兼顾了与router、dispatchManager之类的组件的融合，也能够有较高的性能，适用于没有宿主机权限，性能要求中等的场景。一旦下沉到tcp层，**语义变弱**，决策将会不够灵活。
3. tcp层，劫持系统调用，直接劫持glib里面的函数指针指向我们动态链接库中实现的函数，性能最高，但是无负载均衡、动态扩容等特点，需要在改变宿主机的系统调用，client端不再可信，容易被攻击者篡改。

### 测试配置说明

| 编号 | 配置                     | 说明                                                         |
| ---- | ------------------------ | ------------------------------------------------------------ |
| A0   | Baseline（直连后端）     | 不经过任何代理，Go HTTP Client 直接连接 boltdb 服务，作为性能天花板参照 |
| B0   | Mode 2（SOCKS5 + gRPC）  | 流量经 SOCKS5 → Client → gRPC → Server → 后端，包含完整 gRPC 协议栈 + DispatchManager + OutputBuffer + Sender + Waiter 链路 |
| C0   | Mode 3（劫持且TLS 直连） | 流量经 TLS 握手 → 帧协议 → Server（指纹校验 + Router + Decision Engine）→ 连接池转发 → 后端，同步决策，不经过异步队列 |

------

### GET 操作（读）

> 后端为 boltdb 内存读取，延迟极低（<0.5ms），代理开销占比被放大。这是代理开销的**最坏情况**。

| 配置            | 并发 1    | 并发 5 | 并发 10 | 并发 20 | 并发 50 |
| --------------- | --------- | ------ | ------- | ------- | ------- |
| **A0 Baseline** | 2,916 QPS | 10,655 | 15,013  | 20,162  | 24,845  |
| **B0 Mode 2**   | 358       | 792    | 798     | 791     | 782     |
| **C0 Mode 3**   | 1,035     | 2,594  | 2,667   | 2,690   | 2,683   |

| 配置            | 并发 1 P50（中位数时延） | 并发 5 P50 | 并发 10 P50 | 并发 20 P50 | 并发 50 P50 |
| --------------- | ------------------------ | ---------- | ----------- | ----------- | ----------- |
| **A0 Baseline** | 338µs                    | 450µs      | 621µs       | 888µs       | 1,567µs     |
| **B0 Mode 2**   | 2,726µs                  | 6,123µs    | 12,434µs    | 24,366µs    | 60,872µs    |
| **C0 Mode 3**   | 949µs                    | 1,935µs    | 3,697µs     | 7,360µs     | 18,426µs    |

> **注释：**
>
> - **A0 并发 1（2,916 QPS）** — 单连接直连后端，无并发竞争，反映单请求的最低延迟（P50=338µs）。QPS 不高是因为只有一个连接串行发请求。
> - **A0 并发 50（24,845 QPS）** — 50 个并发连接直连后端，Go HTTP Client 连接池复用，boltdb 并发读无锁竞争，QPS 随并发线性增长。
> - **B0 并发 1（358 QPS）** — Mode 2 单连接，gRPC 协议栈（HTTP/2 + protobuf + 拦截器链）的固有开销约 2.7ms/请求，是 Baseline 的 8.2 倍。
> - **B0 并发 10~50（QPS 稳定在 ~790）** — Mode 2 的 QPS 不随并发增长，瓶颈在 gRPC stream 的串行处理和 DispatchManager 的 Worker 消费速率。
> - **C0 并发 1（1,035 QPS）** — Mode 3 单连接，TLS + 帧协议 + 安全决策的固有开销约 0.95ms/请求，是 Baseline 的 2.8 倍。
> - **C0 并发 10~50（QPS 稳定在 ~2,680）** — Mode 3 的 QPS 同样在并发 10 时饱和，每条 TLS 连接内请求串行处理，增加并发只增加连接数，单连接吞吐不变。

### SET 操作（写）

> boltdb 的写操作触发磁盘 fsync，后端延迟约 1-10ms，代理开销占比相对较小。

| 配置            | 并发 1 | 并发 5 | 并发 10 | 并发 20 | 并发 50 |
| --------------- | ------ | ------ | ------- | ------- | ------- |
| **A0 Baseline** | 694    | 918    | 898     | 917     | 911     |
| **B0 Mode 2**   | 250    | 624    | 633     | 627     | 586     |
| **C0 Mode 3**   | 480    | 873    | 869     | 865     | 863     |

| 配置            | 并发 1 P50 | 并发 5 P50 | 并发 10 P50 | 并发 20 P50 | 并发 50 P50 |
| --------------- | ---------- | ---------- | ----------- | ----------- | ----------- |
| **A0 Baseline** | 1,396µs    | 5,281µs    | 10,744µs    | 21,293µs    | 53,983µs    |
| **B0 Mode 2**   | 3,901µs    | 7,734µs    | 14,374µs    | 31,305µs    | 81,403µs    |
| **C0 Mode 3**   | 2,025µs    | 5,550µs    | 11,179µs    | 22,609µs    | 56,871µs    |

> **注释：**
>
> - **A0 并发 1~50（QPS 稳定在 ~900）** — boltdb 单文件写入，fsync 是瓶颈，增加并发无法提升吞吐。这是测试环境的固有限制。
> - **B0 并发 1（250 QPS）** — gRPC 协议栈开销叠加 boltdb 写延迟，QPS 最低。
> - **C0 并发 5~50（QPS 稳定在 ~865）** — Mode 3 的写操作 QPS 与 Baseline 差距极小（865 vs 911），因为后端 fsync 延迟（~10ms）主导了总延迟，代理增加的 ~1ms 开销被掩盖。
> - **三个模式的 SET QPS 都不随并发增长** — 瓶颈完全在 boltdb 磁盘同步，不在代理。

| 配置        | 操作 | QPS    | P50      | P95      | P99      |
| ----------- | ---- | ------ | -------- | -------- | -------- |
| A0 Baseline | GET  | 15,013 | 621µs    | 1,076µs  | 1,547µs  |
| B0 Mode 2   | GET  | 798    | 12,434µs | 15,481µs | 24,770µs |
| C0 Mode 3   | GET  | 2,667  | 3,697µs  | 4,395µs  | 4,930µs  |
| A0 Baseline | SET  | 898    | 10,744µs | 12,768µs | 17,366µs |
| B0 Mode 2   | SET  | 633    | 14,374µs | 21,640µs | 26,745µs |
| C0 Mode 3   | SET  | 869    | 11,179µs | 13,102µs | 17,990µs |

>  **注释：**
>
> - **GET 并发 10** — Mode 3 的 P50（3.7ms）是 Mode 2（12.4ms）的 **30%**，QPS 是 Mode 2 的 **3.3 倍**。
> - **SET 并发 10** — Mode 3 的 P50（11.2ms）与 Baseline（10.7ms）仅差 0.5ms，说明写操作的代理开销几乎可忽略。
> - **P99 尾部延迟** — Mode 2 的 P99 在高并发下显著偏高（并发 50 时 134ms），原因是 gRPC 流的排队效应。Mode 3 的 P99 表现更稳定。

### Mode 3(直连) vs Mode 2（gRPC）

| 操作                | 并发 1 | 并发 5 | 并发 10 | 并发 20 | 并发 50 |
| ------------------- | ------ | ------ | ------- | ------- | ------- |
| **GET（QPS 倍数）** | 2.9x   | 3.3x   | 3.3x    | 3.4x    | 3.4x    |
| **SET（QPS 倍数）** | 1.9x   | 1.4x   | 1.4x    | 1.4x    | 1.5x    |

GET 操作 Mode 3 稳定比 Mode 2 快 **3.3 倍**。差距来源是 gRPC 协议栈（HTTP/2 帧、protobuf 序列化、拦截器链）与 TLS + 4 字节帧的固有开销差异。SET 操作差距较小（1.4x），因为后端 fsync 延迟主导，代理开销被掩盖。

### Mode 3 vs Baseline

| 操作                | 并发 1 | 并发 5 | 并发 10 | 并发 20 | 并发 50 |
| ------------------- | ------ | ------ | ------- | ------- | ------- |
| **GET（开销占比）** | 35.5%  | 24.3%  | 17.8%   | 13.3%   | 10.8%   |
| **SET（开销占比）** | 69.2%  | 104.9% | 103.4%  | 105.8%  | 105.5%  |

GET 操作中 Mode 3 的代理开销在并发 1 时为 35.5%，随并发增加降至 10.8%。SET 操作开销超过 100% 是因为 boltdb 写延迟波动较大，不是代理导致的。

### 结论

在相同安全组件配置下，Mode 3 的 GET 操作 QPS 始终是 Mode 2 的 3.3 倍，SET 操作 QPS 是 Mode 2 的 1.4 倍。差异来源是协议栈开销 — Mode 2 的 gRPC 链路（HTTP/2 + protobuf + 拦截器链 + DispatchManager + OutputBuffer + Sender + Waiter）每请求经过六个异步组件，Mode 3 仅经过 TLS + 帧协议 + 同步决策 + 连接池转发四个环节。

测试环境中后端延迟极低（boltdb ~0.5ms），代理固定开销（Mode 3 ~1.5ms，Mode 2 ~5.5ms）被放大到 300%-1100%。在真实生产场景中，后端延迟通常在 5-200ms 范围内，Mode 3 的代理开销占比降至 7%-30%，Mode 2 降至 27%-110%。对于外部 API 调用（延迟 >50ms），两种模式的代理开销均可忽略。

**在写密集的场景，或者说IO密集的场景，gRPC连接和TCP直连差距较小。在性能敏感的场景（如网络开销比较小），TCP直连的性能会有显著提高。性能的差距取决于所负责的业务。**
