```
输入进来
  → 硬规则先过滤一轮（你已经有 Router 了，这个就是加几条硬规则的事）
  → 软规则算一个风险分（加权求和，不复杂）
  → 乘一个置信度（连接次数越多值越大，很直觉）
  → 跟代价比一下（谁大选谁，一比就知道）
  → 输出决策
```

其实这个决策引擎本质上就是一个函数，在内部进行不断的计算罢了，这样看其实一步步写不会非常难的。

## 一、先对齐现状：决策引擎目前卡在哪里



从你现有的架构来看，有三个已知问题：



1. 1.**Decision Engine 是硬编码的**，完全针对 IDV 场景的请求编写，不具备通用性。
2. 2.**Portrait Engine 原型已建好**（连接摘要的生成、存储、查询），但历史画像如何转化为决策依据还没想清楚——阈值怎么定、时间衰减怎么做、跨规则怎么联动。
3. 3.**Router 已经做了风险分级**（L0-L3），L0 穿透不进决策引擎，这是对的。但 L1-L3 进来之后，决策引擎怎么处理这些不同等级的任务，目前没有系统性的模型。



所以决策引擎要解决的核心问题是：**给定一个请求的实时行为 + 历史画像 + 风险等级，输出一个决策（ALLOW / AUDIT / ALERT / BLOCK / BLACKLIST），并且这个决策过程是可配置、可解释、可演进的。**

## 三、核心建模：从"规则匹配"到"风险评估"



你现有架构里 Worker 的处理流水线是：指纹匹配 → 加载画像 → Decision Engine 逐条匹配规则。这个"逐条匹配规则"就是我们要改造的核心。



### 3.1 为什么不能继续用硬编码规则



硬编码规则（R01-R09 那种）的问题之前讨论过：权重是拍脑袋的，阈值是拍脑袋的，不同维度的分数不可比。但还有一层更根本的问题：**规则之间没有关联**。R01（连接时间异常 +10）和 R04（删除操作比例高 +20）各自独立打分然后加起来，这假设了各维度是独立的，但实际上它们之间有交互效应——同样是"非典型时间连接"，如果设备一直有凌晨运维的习惯，这个信号几乎没有风险；如果设备从未在凌晨连接过，这个信号的风险权重应该很高。



### 3.2 三层决策模型



我建议的建模思路是把决策拆成三个层次，对应你之前提的三根支柱：



**第一层：硬规则过滤（Filter）**



这是你 Router 已经在做的事情，我认为可以进一步明确：硬规则不应该只做风险分级，还应该直接做阻断判断。具体来说：



- 已知恶意模式 → 直接 BLOCK（比如：同一 UUID 从两个不同 IP 同时连接）
- 合规性要求 → 直接 BLOCK（比如：非注册设备、证书过期）
- 这些规则不需要打分，命中即执行，耗时在微秒级



这一层的本质是**确定性决策**，不涉及概率和代价。



**第二层：风险因子评估（Score）**



这是核心改造部分。每一条软规则不再输出一个固定分数，而是输出一个**风险因子**，包含三个属性：



- **是否触发**（bool）：这条规则在当前请求下是否生效
- **触发强度**（0-1）：偏离正常行为的程度
- **权重**（0-1）：这条规则在整体评估中的重要性占比



关键变化是"触发强度"不再是固定的，而是根据**偏离程度**动态计算的。举个例子：



text

```
规则：写操作占比异常
  历史基线：10%
  当前请求的写操作占比：15%
  强度 = (15 - 10) / 10 = 0.5

  如果当前是 30%：
  强度 = (30 - 10) / 10 = 2.0 → clamp → 1.0
规则：写操作占比异常
  历史基线：10%
  当前请求的写操作占比：15%
  强度 = (15 - 10) / 10 = 0.5

  如果当前是 30%：
  强度 = (30 - 10) / 10 = 2.0 → clamp → 1.0
```



所有触发的风险因子加权合成为**综合风险概率 p**（归一化到 0-1）。



**第三层：代价敏感决策（Decide）**



有了风险概率 p，决策不是"p > 阈值就拦"，而是比较两种错误的期望代价：



text

```
如果放行：期望代价 = p × 误放代价（CostFP）
如果阻断：期望代价 = (1-p) × 误拦代价（CostFN）

选择期望代价更小的行动
如果放行：期望代价 = p × 误放代价（CostFP）
如果阻断：期望代价 = (1-p) × 误拦代价（CostFN）

选择期望代价更小的行动
```



这一层的建模核心是**代价模型**——误放和误拦的代价不是固定值，而是根据请求上下文动态计算的。

## 五、置信度的建模



置信度是你的架构中非常有价值的一个概念，它解决的是"数据不足时怎么办"的问题。



### 5.1 置信度的本质



置信度回答的是："我们有多少数据来做这个判断？"同样的风险概率 0.6，如果置信度只有 0.3（新设备，数据不足），我们应该倾向于 ALERT 而非 BLOCK；如果置信度是 0.9（老设备，画像充分），我们可以更果断地 BLOCK。



### 5.2 置信度的计算



置信度由以下因素决定：



text

```
基础置信度：
  有证书 + 指纹匹配通过 → 0.3（基本的数据来源可信）

画像丰富度：
  连接次数 → 连接越多，画像越可信
  50 次连接 → 满分贡献

基线是否建立：
  有基线 → 额外贡献
  无基线 → 不贡献

HTTP 语义数据是否可用：
  模式一/二有完整的 HTTP 语义 → 额外贡献
  模式三只有 TCP 流量 → 不贡献

  有证书 + 指纹匹配通过 → 0.3（基本的数据来源可信）

画像丰富度：
  连接次数 → 连接越多，画像越可信
  50 次连接 → 满分贡献

  有基线 → 额外贡献
  无基线 → 不贡献

  模式三只有 TCP 流量 → 不贡献
```



### 5.3 置信度对决策的影响



置信度直接参与风险概率的调整：



text

```
调整后的风险 p' = p × confidence
调整后的风险 p' = p × confidence
```

这意味着：

## 八、关于原型的取舍

你提到"很多方案不需要全部加到原型里面"，我建议按优先级分层：



### 第一优先级（原型必须有）：



1. 1.**三层决策结构**（Filter → Score → Decide）
2. 2.**风险因子模板**（通用因子，权重可配置）
3. 3.**基础置信度计算**（基于连接次数和基线建立状态）
4. 4.**代价敏感决策**（基础代价模型，路径级别和时间级别的调整因子）
5. 5.**决策结果带完整上下文**（RiskScore、Confidence、Factors 列表、代价比较结果），方便后续调试和优化



### 第二优先级（原型可以有，但简化）：



1. 6.**告警分级**（根据 riskScore 分低中高优先级）
2. 7.**基线漂移检测**（用简单的移动平均，不做复杂的统计模型）



### 第三优先级（原型不做，生产化再考虑）：



1. 8.反馈回路（需要运维团队配合）
2. 9.时间序列异常检测（需要足够的历史数据）
3. 10.多实体关联（需要全局视角的数据结构）
4. 11.影子模式（需要双引擎架构）
5. 12.对抗鲁棒性（阈值随机化等）

## 九、与现有架构的衔接



决策引擎的改造需要考虑与现有组件的兼容：



1. 1.**Router 不变**：Router 继续做 L0-L3 风险分级，L0 继续穿透。决策引擎只处理 L1-L3 的任务。
2. 2.**Fingerprint Tree 不变**：指纹不匹配直接 BLOCK，这是硬规则，不进入决策引擎。
3. 3.**Portrait Engine 扩展**：现有的 ConnectionSummary 和 UserProfile 结构基本够用，可能需要增加时间序列数据（用于趋势分析）和置信度相关的元数据。
4. 4.**Worker 流水线微调**：Worker 的处理步骤从"逐条匹配规则"改为"风险因子评估 → 代价计算 → 决策"，但整体流程不变。
5. 5.**模式三的特殊考虑**：模式三不走 DispatchManager，是同步转发。决策引擎本身应该是无状态的纯函数（输入上下文 → 输出决策），这样三种模式可以共享同一个决策逻辑，只是调用方式不同。
