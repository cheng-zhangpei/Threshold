## 一、项目概述

**Threshold** 是一个安全代理系统，部署于用户终端设备与 OpenStack 云平台之间，作为透明中间层提供设备身份校验、用户行为评估与访问控制决策能力。系统遵循零信任原则，所有安全决策逻辑集中在服务端执行，客户端仅承担数据采集与传输职能。当前采用 Go 语言实现，处于原型阶段。

系统由两部分组成：

- **Threshold Client（客户端）**：部署在用户终端，负责流量重定向、设备指纹附加、行为事件采集，通过 gRPC 加密通道将请求转发至服务端。
- **Threshold Server（服务端）**：部署为独立安全网关，负责流量解密、设备指纹校验、静态风险分级、行为分析、用户画像构建与访问控制决策。

## 二、系统概述

### 2.1 架构

Threshold 位于 IDV Client 与 OpenStack 云平台之间，所有从终端发往云平台的流量经由 Threshold Client 截获后，通过 gRPC 加密通道转发至 Threshold Server，由服务端完成安全校验和访问控制决策后，将通过的消息传递给 OpenStack。

```
IDV Client  →  Threshold Client  →  Threshold Server  →  OpenStack
              （数据采集+传输）       （安全决策+控制）       （业务平台）
```

### 2.2 客户端组件

客户端部署于用户终端设备，由三个独立运行的组件构成：

| 组件                    | 职责                                                         |
| ----------------------- | ------------------------------------------------------------ |
| 代理核心（LocalProxy）  | 客户端的主进程，同时作为 gRPC Server 监听 :9090 接收 IDV Client 连接，和 gRPC Client 连接 Threshold Server。负责透传 EstablishConnection、ProxyStream、CloseConnection 三个核心 RPC 及 RegisterDevice/UnregisterDevice/ListDevices 三个管理员接口，每个连接维护独立的 connState，连接关闭时清理。 |
| 行为采集器（Collector） | 每个连接独立持有一个实例，随 connState 创建和销毁。记录当前连接中所有操作事件（操作类型+时间戳），生成格式为 `total:write:read:lastOp` 的行为摘要标签（OpTag）。 |
| 流量重定向（Redirect）  | 可选模块，提供 proxychains（命令行）、PAC 脚本（浏览器）、LD_LIBRARY_PATH（自研应用）三种重定向方案。当前架构下 IDV Client 通过 gRPC 主动连接客户端，该模块非必须（暂未实现） |

### 2.3 服务端组件

服务端部署于独立服务器，承担安全代理系统最核心的职责。由六个组件构成，全部运行于同一进程内，通过内存中的 channel 和队列通信：

| 组件                         | 职责                                                         |
| ---------------------------- | ------------------------------------------------------------ |
| gRPC 接入层                  | 服务端唯一网络入口，承担 TLS 终止、请求反序列化与令牌桶基础限流。通过 gRPC 拦截器链注入横切关注点，不做业务判断 |
| Router                       | 读取请求的操作标识（HTTP 方法+路径），查 OPERATION_RISK 静态映射表，判定操作风险等级（L0/L1/L2/L3）。L0 直接穿透至 OutputBuffer，L1 及以上入队交由 Worker 处理 |
| DispatchManager + WorkerPool | 调度中枢，管理 Worker 生命周期与弹性伸缩（默认 2-64 个 Worker）。Worker 无状态，通过竞争消费共享消息队列。支持内存队列满时溢出至 bbolt 冷存储（后续可换统一存储系统），空闲时批量回捞 |
| 指纹匹配引擎                 | 对请求中携带的六维设备指纹进行快速校验。采用六层 Hash 树结构，每层对应一个指纹维度，从 UUID 层开始逐层匹配，最坏情况时间复杂度 O(7)。持久化采用 WAL + bbolt 两层方案 |
| 用户画像与决策引擎           | 为每个用户维护连接粒度的画像数据（ConnectionContext 实时事件 + ConnectionSummary 历史摘要 + UserProfile 聚合评分）。决策引擎以声明式规则（DECISION_RULES）评估请求，规则按严重程度排序，命中第一条即停 |
| OutputBuffer + AlertQueue    | 通过校验的消息暂存于 OutputBuffer，通知下游 OpenStack 侧拉取；被阻断的消息和告警信息暂存于 AlertQueue，推送至管理员监控端。两个组件共同解耦上下游 |

### 2.4 系统边界

| 能力                                    | 说明                         |
| --------------------------------------- | ---------------------------- |
| Router 入口 + L0-L3 静态分级            | 按操作类型判定风险等级       |
| DispatchManager + Worker 弹性伸缩       | 按负载动态调整并发处理能力   |
| 六层 Hash 树指纹匹配 + WAL + bbolt      | 设备合法性校验与持久化       |
| 以连接为单位的用户画像 + DECISION_RULES | 行为评估与访问控制决策       |
| OutputBuffer + gRPC Notify              | 通过消息暂存与下游通知       |
| AlertQueue + 设备拉黑                   | 告警管理与设备黑名单         |
| mTLS 双向认证                           | 加密传输通道                 |
| 跨设备关联画像接口                      | 接口定义（具体算法后续迭代） |

## 三、详细设计

### 3.1 客户端（Threshold Client）

#### 3.1.1 整体架构

客户端核心是一个 gRPC 代理网关，部署在用户终端设备上，作为 IDV Client 与 Threshold Server 之间的透明中间层。客户端实现了 `pb.SecurityProxyServer` 接口，同时承担两个角色：面向下游作为 gRPC Server 监听 :9090 接收 IDV Client 连接，面向上游作为 gRPC Client 连接 Threshold Server 透传请求。

客户端遵循三个核心原则：

- **不做安全决策**：客户端处于不可信环境，所有决策逻辑集中在服务端执行，客户端仅承担数据采集与传输职能。即使客户端被攻破，攻击者也无法绕过服务端的安全策略。
- **无状态**：客户端不维护跨连接的状态。每个连接持有一个独立的 Collector 实例，连接断开时随 connState 一起销毁。画像的跨连接聚合由服务端 PortraitStore 统一负责。
- **透传优先**：客户端不解析原始 HTTP 请求的业务含义，不修改请求体，不缓存响应，请求到达后记录事件即转发。

客户端由三个组件构成：

```
client/
├── proxy/proxy.go        — 代理核心（gRPC Server + gRPC Client）
├── collector/collector.go — 行为事件采集器
└── redirect/redirect.go  — 流量重定向工具（可选）
```

三个组件之间的关系：Redirect 将流量导入 Proxy 的监听端口；Proxy 接收请求后调用 Collector 记录事件，然后透传到 Server；Collector 每个连接独立持有，随 connState 生命周期管理。

#### 3.1.2 代理核心（LocalProxy）

**定位与职责**

LocalProxy 是客户端的主进程，承担连接管理、RPC 透传和连接状态维护三项职能。

| 职责        | 说明                                                         |
| ----------- | ------------------------------------------------------------ |
| gRPC Server | 监听 :9090，接收 IDV Client 的 gRPC 连接， IDV Client需要主动链接我们的代理核心 |
| gRPC Client | 连接 Threshold Server（当前为 insecure 连接）                |
| 连接管理    | 每个 EstablishConnection 创建一个 connState，CloseConnection 时清理 |
| RPC 透传    | EstablishConnection / ProxyStream / CloseConnection 三个核心 RPC 直接转发 |
| 管理员转发  | RegisterDevice / UnregisterDevice / ListDevices 三个管理员接口透传到 Server |

**连接状态管理**

每个连接的生命周期由 connState 管理：

```
连接建立 → 创建connState{connID, collector}
连接存续 → collector持续记录事件
连接关闭 → 清理connState，collector随之销毁
```

connState 仅存在于内存中，不持久化。连接异常断开时（如网络中断），connState 在超时后自动清理。

客户端选择 gRPC Server 模式而非透明代理（PAC/proxychains），原因如下：

| 方案             | 优点                                       | 缺点                                                         |
| ---------------- | ------------------------------------------ | ------------------------------------------------------------ |
| 透明代理         | IDV Client 无需修改                        | 需操作网络栈（TUN 设备、iptables），部署复杂，跨平台兼容性差 |
| gRPC Server 模式 | 协议清晰，用户态实现，部署简单，无 OS 依赖 | IDV Client 需主动连接                                        |

选择 gRPC Server 模式的核心理由是：不需要操作宿主机网络栈，IDV Client 只需一个 gRPC 连接即可接入，未来可以同时支持两种模式（gRPC 为主，透明代理为可选）。

#### 3.1.3 行为采集器（Collector）

**定位与职责**

Collector 以每个连接独立的实例运行，负责记录当前连接中的所有操作事件并生成行为摘要标签。Collector 随 connState 创建和销毁，连接关闭时数据随 connState 一起清理。

**接口定义**

| 方法                 | 说明                                   |
| -------------------- | -------------------------------------- |
| Record(method, path) | 记录一次操作事件，提取操作类型和时间戳 |
| Events()             | 返回当前连接中所有事件的副本           |
| EventCount(opType)   | 按操作类型统计出现次数                 |
| TotalEvents()        | 返回当前连接的总事件数                 |
| OpTag()              | 生成行为摘要标签                       |
| Reset()              | 清空事件列表（连接关闭时调用）         |

**以连接作为管理粒度**

- 以连接为粒度进行画像分析，一个连接等于一个会话
- 避免跨连接的事件混淆
- 连接关闭时 collector 数据随 connState 一起清理，无需额外的生命周期管理
- 跨连接的聚合分析由服务端 PortraitStore 统一负责，客户端不做跨连接的状态积累

**连接建立**

### 3.2 服务端（Threshold Server）

#### 3.2.2 gRPC 接入层

**定位与职责**

gRPC 接入层是服务端对外暴露的唯一网络入口，承担三项职能：TLS 终止、请求反序列化与令牌桶基础限流。拦截器链的执行顺序为：限流检查 → panic恢复 → 请求日志。

接入层仅负责流量能不能进来（TLS 验证、限流），不负责"进来之后怎么办"（操作风险判定、行为评估）。保持接入层轻量，将所有业务判断下沉至 Router 和 Worker 层，接入层仅作为协议边界和流量闸门。

**令牌桶限流**

接入层实现令牌桶限流，作为全局流量的第一道闸门。限流粒度为全局限流（不区分用户），更细粒度的限流由 Router 和决策引擎在业务层实现。限流的目标不是精确控制，而是防止瞬时流量将下游 DispatchManager 和 WorkerPool 打穿。

- 默认参数：每秒 1000 个请求，桶容量 2000
- 超限时返回 gRPC `ResourceExhausted` 状态码
- 生产阶段根据实际流量监控数据调整参数

**拦截器链**

通过 gRPC 拦截器实现三层横切关注点，而非在业务代码中硬编码：

| 层级     | 功能                               | 处理方式                           |
| -------- | ---------------------------------- | ---------------------------------- |
| 限流检查 | 全局令牌桶限流                     | 超限返回 ResourceExhausted         |
| 恐慌恢复 | 捕获 Worker panic                  | 返回 Internal 状态码，记录堆栈日志 |
| 请求日志 | 记录每次请求的方法、耗时、对端地址 | 结构化日志输出                     |

**mTLS 双向认证**

原型阶段采用单向 TLS：服务端持有证书和私钥，客户端跳过证书验证。生产阶段升级为双向 TLS，客户端也需持有由 CA 签发的客户端证书。被拉黑的设备通过吊销其客户端证书实现永久拒绝，无需依赖应用层的黑名单检查。

#### 3.2.3 Router

**定位与职责**

Router 接收 gRPC 接入层反序列化后的请求消息，根据 gRPC 方法路径和原始 HTTP 请求中的方法与路径，查 OPERATION_RISK 静态映射表，判定操作风险等级（L0/L1/L2/L3）。

**风险等级定义**

| 等级 | 触发条件                 | 处理方式                               |
| ---- | ------------------------ | -------------------------------------- |
| L0   | 只读查询（GET 操作）     | 直接穿透至 OutputBuffer，不入队        |
| L1   | 写操作（POST/PUT/PATCH） | 入队至 DispatchManager，兜底动作 AUDIT |
| L2   | 高风险操作（DELETE）     | 入队至 DispatchManager，兜底动作 ALERT |
| L3   | 极高风险操作（保留）     | 入队至 DispatchManager，兜底动作 BLOCK |

**处理流程**

```
Router.route(message)
    │
    ├── ① 提取操作标识：从原始 HTTP 请求中解析 method + path，组装为 op_key
    │      例如："GET /api/cloud/public/images"
    │
    ├── ② 查 OPERATION_RISK 静态映射表
    │      支持通配符路径匹配（正则表达式预编译）
    │      例如："/api/local/images/*/sync" 编译为正则后匹配
    │
    ├── ③ 未知操作默认定级为 L1
    │
    ├── ④ L0 → 调用 dispatch.direct_pass(message) 直接穿透
    │
    └── ⑤ L1+ → 调用 dispatch.enqueue(message, risk_level) 入队等待 Worker 处理
```

**路径模糊匹配**

OPERATION_RISK 映射表中的路径可能包含通配符。采用正则表达式预编译方式：映射表中的路径模式在初始化时编译为正则表达式（如 `/api/local/images/*/sync` 编译为 `^/api/local/images/[^/]+/sync$`），运行时直接匹配，避免每次请求都做字符串解析。

对于映射表中不存在的操作，Router 将其默认定级为 L1（写操作，需审计）而非 L0（直接放行）。基于安全原则的保守选择：未知操作应被审慎对待，交由 Worker 进一步分析，而非直接穿透。代价是新增 API 端点时需要同步更新映射表，否则新端点的请求都会进入 Worker 队列。系统目前认为这个代价可接受，因为新增 API 端点本身就是低频操作。

#### 3.2.4 DispatchManager 与 WorkerPool

**定位与职责**

DispatchManager 是服务端的调度中枢，位于 Router 和决策引擎之间。职责是接收 Router 分级后的任务，通过 WorkerPool 弹性伸缩处理，并保证在高并发下的背压可控。

**核心组件**

| 组件            | 职责                                                |
| --------------- | --------------------------------------------------- |
| DispatchManager | 调度中枢，管理任务队列、Worker 生命周期、溢出持久化 |
| Worker          | 无状态处理单元，从队列竞争消费任务                  |
| TaskStore       | 溢出持久化层，封装 bbolt 读写                       |

**Worker 内部处理流水线**

```
1. 从共享队列读取任务（竞争消费）
2. 构建 ConnectionContext（注入当前操作事件）
3. 指纹 Hash 树匹配
   ├─ 匹配失败 → 返回 BLOCKED + 写入 AlertQueue
   └─ 匹配成功 → 继续
4. 加载用户画像
   ├─ ConnectionContext（本次连接的实时事件）
   └─ PortraitStore.GetHistory（最近 N 次连接的历史摘要）
5. 决策引擎逐条匹配 DECISION_RULES
6. 通过 task.ResultCh 同步返回决策结果给调用者
```

**弹性伸缩策略**

DispatchManager 维护一个后台协程 monitor_loop，每 5 秒检查一次 WorkerPool 状态：

| 参数                   | 默认值 | 说明                     |
| ---------------------- | ------ | ------------------------ |
| MinWorkers             | 2      | 最小保底 Worker 数       |
| MaxWorkers             | 64     | 最大上限                 |
| ScaleUpThreshold       | 100    | 队列深度超过此值触发扩容 |
| ScaleUpStep            | 4      | 每次扩容增加的 Worker 数 |
| ScaleDownThreshold     | 20     | 队列深度低于此值触发缩容 |
| ScaleDownStep          | 2      | 每次缩容减少的 Worker 数 |
| HealthCheckIntervalSec | 5      | 巡检间隔                 |

Worker exit通过 atomic.Bool.CompareAndSwap 实现无锁标记。exit是软的——Worker 在处理完当前任务后才检查exit标记，不会中断正在处理的任务，保证不会丢失正在执行的请求。

monitor_loop每 5 秒检查一次队列深度，按以下规则执行：

```
检查队列深度
    │
    ├── 队列深度 > ScaleUpThreshold 且 Worker 数 < MaxWorkers → 扩容：新增 ScaleUpStep 个 Worker
    │
    ├── 队列深度 < 20 且 Worker 数 > ScaleDownStep → 缩容：标记 ScaleDownStep 个最久空闲的 Worker exit
    │
    └── 其他情况 → 不动
```

例子：

```
t=0s    2 个 Worker，task=0
        突然 200 个请求到达

t=5s    monitor 检查：task=150 > 100 → 扩容到 6 个 Worker

t=10s   monitor 检查：task=130 > 100 仍然积压 → 再扩容到 10 个 Worker

t=15s   队列消化完毕，monitor 检查：task=0 < 20 → 缩容到 8 个

t=20s   task 仍空闲 → 缩容到 6 个

t=25s   继续缩 → 4 个
t=30s   继续缩 → 2 个（MinWorkers，不再缩）
```

**背压与溢出机制**

当请求量突增导致内存队列被打满时，不是直接拒绝或阻塞等待，而是将任务溢出到 bbolt 冷存储，队列空闲后再批量回捞，这个机制很重要，因为我们之前的router也是一个消费队列，如果后面dispatchManager的异步队列不保证不阻塞，一旦出问题会导致router的异步队列级联奔溃使得整个server被打崩掉。

| 阶段     | 处理方式                                                     |
| -------- | ------------------------------------------------------------ |
| 队列未满 | 任务投递到内存队列，调用者阻塞等待 Future（这在go里面的表现就是一个chan，如果任务完成就会往这个chan里面发信息，是go语言异步等待队列的一种常见的实现方式） 完成 |
| 队列满   | 任务写入 bbolt，原始调用者的 Future 保留在内存 pending map 中 |
| 回捞     | monitor_loop 检测到队列深度 < ReloadSize，从 bbolt 批量读取，从 pending map 取出原始 Future 组装任务投递队列 |
| 极端场景 | bbolt 也满 → Overflow 返回错误 → Router 收到 THROTTLE。所有路径都有非阻塞出口，不会死锁 |

关键设计：溢出时 resultCh（Future）不丢失。即使任务经过"内存 → bbolt → 内存"的完整冷路径，调用者仍然能拿到决策结果。

**同步结果返回 vs 异步 Fire-and-Forget**

Worker 处理完成后通过 Future 将决策结果同步返回给 Router，Router 再返回给客户端。选择同步返回的原因是客户端需要立即知道请求是否被放行（如被拒绝时提示用户）。替代方案是 fire-and-forget——Worker 处理完成后只投递到 OutputBuffer，客户端不等待结果。同步返回的代价是 Router 调用 enqueue 后会阻塞等待，但 L0 穿透不入队不阻塞，L1 审计类操作处理速度快，真正耗时的操作（如镜像扫描）由 CICD 模块独立执行，Worker 只需触发扫描任务即可立即返回。

#### 3.2.5 指纹匹配引擎

**定位与职责**

指纹匹配引擎对请求中携带的六维设备指纹进行快速校验，判断该设备是否为已注册的合法设备。指纹匹配在决策引擎之前执行——如果指纹不匹配，请求直接 BLOCKED，决策引擎不会运行。这是一个硬性前置校验，不作为决策规则。

**六层 Hash 树结构**

```
Level 0: UUID 层      — 设备唯一标识（128 位）
Level 1: OS 层        — 操作系统类型（linux/windows/macos）
Level 2: IP 层        — 客户端 IP 地址
Level 3: Port 层      — 源端口
Level 4: Protocol 层  — 协议类型
Level 5: Reserved 层  — 保留字段（当前为空，预留扩展）
```

匹配算法从 Level 0 开始，每层取出对应维度的值计算哈希，在该层的哈希表中查找。如果当前维度的值为 null（该字段缺省），则跳过该层继续下一层。如果任意一层查找失败，判定为指纹不匹配。最坏情况时间复杂度 O(7)。

**null 节点与缺省匹配**

指纹的某些维度可能缺失（如设备未上报操作系统类型）。缺省字段对应的位置不创建子节点分支，而是直接跳过，继续匹配下一层。

这一设计的权衡在于：跳过某个维度会降低匹配的精确度。本文认为 UUID 本身的唯一性已经足够强（128 位标识符，重复概率接近零），后续维度更多是作为辅助校验而非主标识符，因此单个维度缺失不会显著降低安全性。生产阶段可以通过配置策略要求至少匹配 N 个维度（如至少匹配 UUID + 另外两个维度），对缺省匹配的容忍度进行灵活控制。

**类似于B 树防退化**

每层的哈希 bucket 设置大小阈值（如 1024 个条目），超过阈值时按 B 树规则进行节点分裂。防止极端情况下单个 bucket 退化为线性查找，保证每层的查找始终为 O(log N)。退化主要可能发生在 IP 层（多个设备可能来自同一 IP 段）和 Port 层（常用端口有限），B 树分裂策略对这两层的保护意义最大。

**持久化设计**

指纹索引的持久化采用 WAL + bbolt 两层方案：

| 流程 | 步骤                                                         |
| ---- | ------------------------------------------------------------ |
| 写入 | ① 将指纹变更操作写入 WAL 日志（顺序写，保证崩溃安全）→ ② 异步将 WAL 中的操作应用到 bbolt 中的 Hash 树索引 → ③ 定期生成全量快照 → ④ 快照成功后截断 WAL |
| 读取 | ① 正常情况从内存中的 Hash 树索引读取 → ② 进程重启后从 bbolt 加载索引到内存 → ③ 若 bbolt 损坏，从最近一次快照 + WAL 重放恢复 |

#### 3.2.6 用户画像存储（PortraitStore）

**定位与职责**

PortraitStore 为每个用户维护一份画像数据，记录该用户历次连接的行为摘要。画像数据以连接为粒度存储——每次连接结束时生成一份连接摘要并追加到该用户的画像历史中。决策引擎在处理新请求时同时读取 ConnectionContext（本次连接的实时事件）和 DB 中的历史画像摘要，综合判定风险。

**数据模型**

PortraitStore 维护三层数据：

| 数据层            | 生命周期     | 存储位置 | 用途                                                         |
| ----------------- | ------------ | -------- | ------------------------------------------------------------ |
| ConnectionContext | 连接存续期间 | 内存     | 记录当前连接的实时事件列表，决策引擎的实时信号               |
| ConnectionSummary | 持久化       | bbolt    | 每次连接结束时生成的快照，记录连接时长、事件计数、触发的规则标志等 |
| UserProfile       | 持久化       | bbolt    | 跨连接的聚合统计，每次连接关闭时增量更新                     |

**ConnectionContext 数据结构**

```
ConnectionContext:
    connection_id: str          # 连接唯一标识
    user_id: str                # 用户标识
    device_uuid: str            # 设备标识
    connected_at: float         # 连接建立时间
    ip: str                     # 客户端 IP
    events: list                # [(op_type, timestamp), ...]
    triggered_flags: list       # 本次连接触发的规则标志
```

ConnectionContext 提供以下查询方法：

| 方法                | 用途                            | 规则依赖        |
| ------------------- | ------------------------------- | --------------- |
| event_counts()      | 按操作类型聚合计数              | R03, R07, R08   |
| write_event_count() | 写操作总数                      | R09, R10        |
| write_ratio()       | 写操作占比                      | R10             |
| last_event(op_type) | 某类操作的最后时间戳            | R08（时序判断） |
| is_off_hours()      | 当前是否非工作时间（0:00-6:00） | R10             |

**ConnectionSummary 数据结构**

连接断开时从 ConnectionContext 提取生成：

```
ConnectionSummary:
    connection_id: str
    user_id: str
    device_uuid: str
    connected_at / ended_at / duration_sec
    ip: str
    event_counts: dict          # {"image.upload": 3, "vm.start": 1, ...}
    flags_triggered: list       # ["alert_admin", "throttle"]
    off_hours_writes: int       # 非工作时间的写操作数
    total_events: int
    write_ratio: float          # 写操作占比
```

**UserProfile 聚合**

UserProfile 在每次连接关闭时增量更新，维护以下聚合字段：

| 字段                     | 说明               |
| ------------------------ | ------------------ |
| TotalConnections         | 历史连接总数       |
| UniqueDevices            | 使用过的不同设备数 |
| UniqueIPs                | 使用过的不同 IP 数 |
| TotalWriteOps            | 累计写操作数       |
| AlertCount               | 累计告警触发次数   |
| OffHoursCount            | 非工作时间连接次数 |
| KnownDevices / KnownIPs  | 已知设备/IP 列表   |
| FirstSeenAt / LastSeenAt | 首次/最近活跃时间  |

**RiskScore 复合评分**

> 这种评分机制其实现在非常简陋，但是原型阶段暂时就不深究如何设计评分机制能够去有效的防御恶意行为

UserProfile.RiskScore() 计算 0.0（安全）到 1.0（高风险）的复合评分：

```
Score = alert_frequency  × 0.40
      + write_ratio      × 0.20
      + device_diversity × 0.20
      + off_hours_ratio  × 0.20

Clamped to [0.0, 1.0]
```

权重选择基于信号可靠性：

| 维度       | 权重 | 理由                                                         |
| ---------- | ---- | ------------------------------------------------------------ |
| 告警频率   | 40%  | 最强信号：系统已判定有问题的直接证据                         |
| 写比例     | 20%  | 中等信号：写操作占比高可能暗示数据破坏，但部分合法工作流（备份、迁移）也是写密集型 |
| 设备多样性 | 20%  | 中等信号：4+ 设备不常见，可能暗示凭证共享或账户入侵          |
| 非工作时间 | 20%  | 弱信号：凌晨工作可疑但不构成证据，时区差异和值班轮换会产生合法的非工作时间活动 |

阈值设定为 0.7，用户需要至少 2 个以上风险维度显著升高才会触发。阈值 0.5 会导致合法多设备用户误触发，阈值 0.9 会遗漏微妙威胁。

**连接断开时的持久化流程**

```
CloseConnection → PortraitStore.on_connection_close(ctx)
    │
    ├── ① 从 ConnectionContext 提取 ConnectionSummary
    ├── ② 事务写入 bbolt（key: userID|timestamp，字典序 = 时间序）
    ├── ③ 自动裁剪：每个用户保留最近 50 条记录
    ├── ④ 增量更新 UserProfile 聚合数据
    └── ⑤ 释放内存中的 ConnectionContext
```

**存储接口抽象**

```
PortraitBackend(ABC):
    get_summaries(user_id, limit) -> list[ConnectionSummary]
    append_summary(user_id, summary)
    is_blacklisted(device_uuid) -> bool
    blacklist_device(device_uuid, reason)
```

初期实现用 bbolt，生产阶段可替换为分布式数据库，上层 PortraitStore 逻辑不变。

**设计决策：以连接为粒度 vs 以时间桶为粒度**

选择以连接为粒度而非时间桶，原因：

| 维度       | 连接方案                     | 时间桶方案                                   |
| ---------- | ---------------------------- | -------------------------------------------- |
| 内存开销   | 仅与并发连接数成正比         | 与总用户数成正比（需维护所有用户的窗口状态） |
| 事件归属   | 连接边界清晰                 | 桶边界处事件归属模糊                         |
| 上下文信息 | 天然携带连接时长、来源 IP 等 | 需额外维护                                   |
| 跨连接分析 | 通过历史摘要中的累计计数捕捉 | 可精确到任意时间窗口                         |

连接方案的代价是决策时无法精确到跨连接的任意时间窗口，但行为异常在单次连接中通常已经表现得足够明显，跨连接的异常通过历史摘要中的 flags_triggered 和累计计数来捕捉。

#### 3.2.7 决策引擎

**定位与职责**

决策引擎是安全代理系统的核心智能组件，被 Worker 调用的函数库。给定一个请求的操作类型、当前连接的行为历史、该用户的历史画像摘要，输出一个明确的决策结果。

**设计原则**

| 原则             | 说明                                         |
| ---------------- | -------------------------------------------- |
| 声明式而非命令式 | 决策规则以配置形式声明，不硬编码在业务逻辑中 |
| 规则可组合       | 多条规则可同时生效                           |
| 上下文感知       | 同一操作在不同上下文中可能得到不同的决策结果 |
| 可审计           | 每次决策的输入、命中规则和输出均可追溯       |

**输入与输出**

```
输入：
  ① RiskLevel — Router 判定的静态风险等级（L0/L1/L2/L3）
  ② ConnectionContext — 当前连接的实时事件
  ③ History — 最近 N 次连接的历史画像摘要（ConnectionSummary 列表，由UserProfile提供）
处理：
  规则引擎逐条匹配 DECISION_RULES
  从上到下，命中第一条即停
输出：
  Decision(action, reason, rule_id)
```

**规则定义（DECISION_RULES）**

规则按四层结构组织，从严重程度最高到最低：

**第一层：设备级拦截（不看行为，只看设备状态）**

| 规则                   | 条件         | 动作         |
| ---------------------- | ------------ | ------------ |
| R01_DEVICE_BLACKLISTED | 设备已被拉黑 | BLOCK_DEVICE |

**第二层：历史连接级拦截（看跨连接的行为模式）**

| 规则                        | 条件                                              | 动作             |
| --------------------------- | ------------------------------------------------- | ---------------- |
| R02_REPEAT_OFFENDER         | 最近 3 次连接都触发过告警                         | BLACKLIST_DEVICE |
| R03_HISTORICAL_HEAVY_DELETE | 最近 5 次连接累计删除操作超过 10 次（含当前连接） | BLACKLIST_DEVICE |
| R04_HISTORICAL_BLACKLISTED  | 历史上被拉黑过，本次要求二次认证                  | REQUIRE_2FA      |
| R05_IP_HOPPING              | 最近 5 次连接来自 3 个以上不同 IP（含当前连接）   | REQUIRE_2FA      |

**第三层：当前连接级拦截（看本次连接内的行为）**

| 规则                  | 条件                                                        | 动作                 |
| --------------------- | ----------------------------------------------------------- | -------------------- |
| R06_BRUTE_FORCE_LOGIN | 当前连接内暴力登录失败超过 5 次                             | BLOCK_LOGIN_10MIN    |
| R07_BULK_DELETE       | 当前连接内删除操作超过 3 次                                 | BLOCK_DEVICE         |
| R08_UPLOAD_THEN_START | 上传镜像后立即启动虚拟机（可疑载荷注入）                    | QUARANTINE_AND_ALERT |
| R09_HEAVY_WRITER      | 当前连接写操作超过 20 次                                    | THROTTLE             |
| R10_OFF_HOURS_PATTERN | 非工作时间 + 写操作超过 5 次 + 历史连接也有非工作时间写操作 | BLOCK_WRITE_OPS      |

**第四层：默认兜底（按 Router 判定的静态风险等级处理）**

| 规则                     | 条件             | 动作                                             |
| ------------------------ | ---------------- | ------------------------------------------------ |
| R99_STATIC_RISK_FALLBACK | 永远命中（True） | 由 risk_level 决定：L1→AUDIT，L2→ALERT，L3→BLOCK |

命中第一条即停，不继续匹配后续规则。

**决策响应动作定义**

| 动作                 | 含义       | 对请求的处理                | 额外副作用                   |
| -------------------- | ---------- | --------------------------- | ---------------------------- |
| ALLOW                | 放行       | 请求转发至 OpenStack        | 无                           |
| AUDIT                | 审计放行   | 请求转发至 OpenStack        | 记录详细日志                 |
| ALERT                | 告警放行   | 请求转发至 OpenStack        | 告警推送至管理员             |
| THROTTLE             | 限速       | 请求转发但限速              | 后续请求增加延迟             |
| BLOCK                | 阻断       | 请求不转发，返回拒绝        | 无                           |
| BLOCK_DEVICE         | 设备阻断   | 该设备后续所有请求拒绝      | 设备标记为临时阻断           |
| BLOCK_LOGIN          | 登录阻断   | 登录请求拒绝，10 分钟后解除 | 定时任务自动解除             |
| BLOCK_WRITE_OPS      | 写操作阻断 | 读操作放行，写操作拒绝      | 仅本次连接生效               |
| REQUIRE_2FA          | 二次认证   | 请求暂挂，要求二次认证      | 通过后正常放行               |
| QUARANTINE_AND_ALERT | 隔离告警   | 镜像存入隔离区              | 触发 CICD 扫描任务           |
| BLACKLIST_DEVICE     | 拉黑设备   | 拒绝所有请求                | 设备永久拉黑 + 锁账户 + 告警 |

**规则配置的可扩展性**

原型阶段规则以代码硬编码。生产阶段建议抽离为 YAML/JSON 配置文件，支持不重启热加载。规则的条件函数抽象为标准比较算子（大于、小于、包含、模式匹配），配置文件中声明算子和参数即可，不需要编写代码中的条件函数。规则的版本化管理可纳入配置中心（如 Consul 或 etcd），实现变更的可追溯和回滚。

#### 3.2.8 OutputBuffer 与 AlertQueue

**定位与职责**

两个组件的设计目标一致：解耦上下游，允许下游按自己的节奏消费。

| 组件         | 消费方       | 推送方式                                                 | 用途                       |
| ------------ | ------------ | -------------------------------------------------------- | -------------------------- |
| OutputBuffer | OpenStack 侧 | 通知+拉取（SubscribeNotify 通知，PullApproved 批量拉取） | 暂存通过校验的消息         |
| AlertQueue   | 管理员监控端 | 主动推送（gRPC stream 推送）                             | 暂存被阻断的消息和告警信息 |

**OutputBuffer**

采用"通知+拉取"模式：有新数据时通过 gRPC SubscribeNotify 通知下游，通知只携带"有新数据"的信号（不包含数据本身），下游收到通知后通过 PullApproved 接口批量拉取数据（一次最多 batch_size 条）。

设计权衡：替代方案是纯轮询（下游每隔固定时间主动拉取）。通知模式的优点是实时性更好，下游不会在没有数据时空跑拉取请求；通知本身只是一次轻量信号传递，开销极小。

**AlertQueue**

告警信息到达后立即通过 gRPC stream 推送给所有订阅的管理员监控端，不等待拉取。设备拉黑操作在 L3 级别阻断时额外触发。

**决策引擎联调（8 个）：**

| 测试                     | 验证内容                                                     |
| ------------------------ | ------------------------------------------------------------ |
| R99 L1 风险兜底          | L1 操作无行为规则命中时 → AUDIT                              |
| R99 L2 风险兜底          | L2 操作无行为规则命中时 → ALERT                              |

| 混合风险                 | 同一队列中混合 L1/L2/L3 任务，各自得到正确的决策结果         |
| 弹性扩缩容下的决策一致性 | 扩容前后的决策结果一致                                       |
| **溢出决策一致性**       | 任务经过"内存→bbolt→回捞→Worker"完整冷路径后，决策结果与热路径完全一致 |

最后一个测试（OverflowPreservesDecision）是整个调度模块最核心的测试——它验证了背压机制不会影响决策的正确性。测试方法是：通过注入延迟（20ms/任务）同时触发溢出和决策引擎联调，对比冷路径和热路径的输出。

**延迟注入设计：**

| 场景                      | 注入延迟   | 目的                            |
| ------------------------- | ---------- | ------------------------------- |
| OverflowToStorage         | 50ms/task  | 队列打满，强制溢出到 bbolt      |
| ScaleUp                   | 100ms/task | Worker 处理慢，队列积压触发扩容 |
| OverflowPreservesDecision | 20ms/task  | 同时触发溢出 + 决策引擎联调     |

延迟注入让测试变慢（每个约 2-4 秒），但这是验证异步行为的必要手段。
